**Attacker Perspective (branch delta only):**
1. Can the new `--waive` flag be used to silently suppress a real
   translation-quality defect from blocking client delivery? By design it
   suppresses a matched error from the pass/fail gate — but each waiver is
   required by the consuming SKILL.md to be listed alongside the delivered
   output with a one-line justification, and the script itself prints every
   waived error under a `WAIVED (treated as pass)` heading rather than
   hiding it — so this is an audited, visible override for reviewed false
   positives, not a silent bypass. No code path lets a waiver suppress
   itself from that visible listing.
2. Can `client_safe` audience mode be used to show a client a different
   verdict than the internal-analyst view would show for the same inputs?
   No — §13.2 states the verdict/classification itself is never altered by
   audience mode; only the arithmetic, apparatus, and cutoff detail
   surrounding it are omitted or relocated. The non-suppressible list (AI-
   disclosure banner, disclaimers, mismatch warnings, footer) is identical
   in both modes.
3. Can an unrecognized or malformed `render.audience_default` value in a
   white-label config cause an unintended, more-permissive rendering state?
   No — the JSON schema constrains the field to two literal enum values;
   the loader's `_build_result` passes through whatever the config file
   contains without additional interpretation, and §13.1 specifies that any
   unrecognized value falls back to the existing default (`internal_analyst`)
   with a visible one-line notice, not a silent or expanded-disclosure
   state.
4. Do the two new `render_gate.py` anchors (`should-i-buy`,
   `score-explainer`) introduce a new way for scaffold/debug text to leak
   into a rendered client report? No new risk beyond the gate's existing,
   already-reviewed fail-open behavior (returns input unchanged if no
   anchor matches) — this diff only extends that same design to two
   additional skills using the same pattern style.
