**A08 Software and Data Integrity:** The headline fix in this diff is a
process-integrity hardening, not a weakening: previously, a hung `claude -p`
rollout subprocess (and any children it spawned, e.g. MCP tool subprocesses)
had no enforced timeout in `run_rollout.sh` and could run indefinitely,
consuming resources and potentially leaving orphaned processes after the
parent script exited. The new code (1) launches the subprocess with
`start_new_session=True`, making it the leader of a new process group; (2)
enforces `proc.wait(timeout=TIMEOUT_SECONDS)` (default 900s, overridable via
`ROLLOUT_TIMEOUT_SECONDS`); (3) on `TimeoutExpired`, calls
`os.killpg(proc.pid, signal.SIGKILL)` — correct because `proc.pid` is the
process-group id when `start_new_session=True` was used, so this reaps the
whole subtree, not just the direct child; (4) deletes the partial/incomplete
output file (`os.unlink(out)`) so a killed rollout cannot be mistaken for a
completed one downstream; (5) exits `124` (the conventional `timeout(1)`
exit code) so callers can detect the timeout case. This closes a
resource-exhaustion / zombie-process gap. PASS.

**Attacker Perspective (branch delta only):**
1. Can the new `--skill` flag be used to load and execute arbitrary Python
   from outside `evals/skills/`? No — the regex `^[A-Za-z0-9_-]+$` rejects
   any `/`, `.`, or other path-traversal character before the value reaches
   `load_spec()`'s path construction; verified empirically against a
   traversal payload.
2. Can a malicious or malformed JSONL task file (`args`, `lang` fields)
   achieve command injection via the rollout subprocess? No — all such
   values are passed as Python `sys.argv` elements into a list-form
   `subprocess.Popen` call, never through a shell. Task files are
   repo-tracked, not user-uploaded, so this is defense-in-depth rather than
   a live threat in this repo's actual usage.
3. Can a hung rollout subprocess now escape the timeout and continue
   consuming resources or leak into a later run? No — `os.killpg` on the
   process group (not just the direct child) reaps subprocess trees started
   by `claude -p` (including any MCP tool child processes), and the partial
   output file is deleted so it cannot be picked up as a valid result.
4. Does the `tier2: None` handling in `aggregate_report.py` silently mask a
   real failure as a pass? No — `below_ceiling` is now `tier1_mean < 1.0 OR
   (tier2_mean is not None AND tier2_mean < 1.0)`; a structural-only spec
   with no tier2 criteria correctly drops out of the tier2 aggregate instead
   of being coerced to `0.0` (which previously would have permanently
   flagged `below_ceiling=True` for any skill without a judge rubric) or
   silently passing. Verified against the new `test_aggregate_report.py`
   cases (all-None, mixed-None, ceiling-flag interaction) — not a security
   issue, but confirms no correctness regression that could mask a real eval
   failure as green.
5. Does the SKILL.md → references/*.md move for white-label-onboard change
   what gets executed or exposed at runtime? No — this is documentation
   reorganization consumed by the LLM reading the skill; no code, no new
