## OWASP Findings Detail (supporting SEC-12) — evals/ new surface

**A01 Broken Access Control:** No new access control surface. The evals/ harness is operator-only CLI tooling with no endpoints, no user roles, no network listeners. PASS.

**A02 Cryptographic Failures:** No new sensitive data stored or transmitted directly. The claude CLI subprocess handles TLS to Anthropic's API; no crypto is implemented in repo code. PASS.

**A03 Injection:**

- Command injection via run_rollout.sh: ARGS (from $1), LANG_ARG (from $2), and TARGET_MODEL (from $4) are all passed as individually quoted strings to claude -p "$PROMPT" and to the rollout call as "$ARGS". PROMPT is constructed as a plain string ("$CMD ${ARGS}") and passed as the -p argument — this is text content sent to claude, not to the shell. No shell metacharacter expansion risk: claude -p receives the string as a single CLI argument. SAFE. The nounset guard for the MODEL_FLAG empty array is correctly handled with ${MODEL_FLAG[@]+"${MODEL_FLAG[@]}"} (Bash conditional expansion, not a bare unquoted array). SAFE.

- Command injection via run_baseline.sh: ARGS and ID are extracted from JSONL via python3 -c inline one-liners using json.loads(). These values are then passed as "$ARGS" and used in heredoc-passed Python scripts (<<'PY') where they appear as subprocess.argv — not interpolated into shell. The heredoc uses a quoted delimiter ('PY') so no variable expansion occurs inside the heredoc body. SAFE.

- Path traversal in eval_spec.load_spec(): the skill parameter (a string like "should-i-buy") is appended to a fixed base path (evals/skills/<skill>/eval_config.py) without sanitization. A skill name containing "../" would traverse directories. However: (a) the skill parameter is either hardcoded in caller code or passed via operator CLI --skill flag; (b) there is no user-supplied or network-supplied skill name path in the production flow. INFORMATIONAL — acceptable risk for an internal developer tool; would become a real finding if this function were exposed via HTTP or to untrusted input.

- Arbitrary code on import via eval_config.py: load_spec() uses importlib.util.exec_module() to import eval_config.py files. This executes arbitrary Python at import time. The files are repo-local, developer-authored, and not user-supplied. This is equivalent to a developer running their own Python modules. INFORMATIONAL — explicitly acceptable given the design constraint that eval_config.py is always repo-local; would become Critical if the path were user-controlled.

- YAML injection: no yaml.load() calls anywhere in evals/. No YAML parsing in the new substrate. PASS.

- Unsafe deserialization: no eval(), exec(), pickle.load(), or yaml.load(unsafe) calls found anywhere in evals/. JSON parsing uses json.loads() throughout. PASS.

- ReDoS in tier1_structural.py regex patterns: the compiled patterns (_REC_PATTERNS, _TREND_PATTERNS, _TASK_RE) are fixed strings with no user-controlled content in the pattern itself. The input matched against them is LLM-generated prose read from disk (eval output), which is attacker-controlled only in the theoretical sense that a prompt-injected model response could craft adversarial input. None of the regex patterns have catastrophic backtracking structure (no nested quantifiers, no alternation within quantifiers). PASS.

- The stderr[:200] truncation in _parse_verdict is a good hygiene measure preventing runaway error output from leaking into judge result files. No key material found in committed judge results. PASS.

**A04 Insecure Design:** No threat model present (unchanged from prior audit). The evals/ substrate is a developer-internal eval harness, not a deployed service. No new attack surface requiring threat modeling has been introduced. LOW priority. Recommend /threat-model before any network-exposed eval endpoint is added.

**A05 Security Misconfiguration:** No debug mode, no default credentials surface. The Anthropic model allowlist guard in run_judge.py (ALLOWED_MODEL_PREFIXES = ("claude-",)) correctly prevents skill content from being sent to non-Anthropic models. The guard is enforced at two levels: (1) run_baseline.sh calls python3 run_judge.py --check-model before any rollouts including in --dry-run mode; (2) judge_one() and grade_tier2() each call check_model() before constructing the subprocess call. Allowlist cannot be bypassed by a well-formed --model flag; would require source code modification. PASS.

**A06 Vulnerable Components:** No new package.json or requirements.txt introduced by evals/. Python graders import only stdlib (json, re, subprocess, pathlib, statistics, importlib.util, dataclasses, collections, argparse). No third-party dependencies added. PASS.

**A07 Authentication Failures:** No auth surface. PASS.

**A08 Software and Data Integrity:** No CI/CD pipeline. No webhook handlers. The evals/ substrate adds no new integrity concerns. PASS.

**A09 Security Logging:** The evals/ graders write to stderr only: perimeter-abort messages (model not on allowlist), pass-rate counts, and rollout cost estimates. No secrets are logged. The stderr[:200] truncation in _parse_verdict prevents potential log injection via claude CLI error output. PASS.

**A10 SSRF:** The evals/ substrate makes outbound calls via claude CLI subprocesses to Anthropic's API. These calls use the operator's shell environment (ANTHROPIC_API_KEY). There is no server-side component and no user-controllable URL input fed to an outbound request. INFORMATIONAL — same classification as the existing web_pdf.py finding in the prior audit; no server exists, so the SSRF threat model does not apply.

**Attacker Perspective (evals/ new surface):**
1. Most sensitive data added: LLM output transcripts (stream.json) in evals/results/ — correctly gitignored. No PII, no credentials. The committed golden/broken fixtures are synthetic AAPL evaluation outputs with no sensitive content.
2. Can ARGS be used to exfiltrate secrets? ARGS is passed as the -p prompt to claude; it can influence LLM behavior but cannot execute shell commands or read files outside claude's sandboxed context. The eval harness is not a remote-execution surface.
3. Can the allowlist guard be bypassed? Only by editing ALLOWED_MODEL_PREFIXES in run_judge.py. No runtime bypass path exists.
4. Can eval_config.py be used to execute malicious code? Only by a developer with repo write access modifying an eval_config.py file — equivalent to arbitrary code execution via any other Python file in the repo. Not an incremental risk.
