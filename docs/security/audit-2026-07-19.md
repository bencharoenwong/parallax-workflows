## Critical Findings Detail

### Finding: Real internal/proprietary identifiers committed as literal test-fixture strings, in a source path the leak-prevention gate never scans — REMEDIATED, independently re-verified

**Severity**: Critical (at time of original finding; now resolved)
**Location**: `skills/_parallax/scripts/test_build_bundle.py`, four test
functions in the working-tree diff (case-variant-leak test, the
allowlist-scoping test, and two allowlist-masking edge-case tests).
**CWE**: CWE-540 (Inclusion of Sensitive Information in Source Code)

### Description (original finding)

The original test coverage for `canary_scan`'s case-insensitivity fix used, as
its "realistic leak" fixture strings, several literal terms that are themselves
real entries on the locally-maintained, gitignored extra-canary list
(`~/.claude/parallax-canary-extra.txt`) — a file whose own header states these
terms must never be copied into any tracked file in this public repo. Confirmed
at the time, and reconfirmed now: `git show HEAD:skills/_parallax/scripts/
test_build_bundle.py` (the committed version) has never contained these terms
— the exposure existed only in the uncommitted working tree and was caught
before any commit or push.

Separately, and still true after remediation (this is a structural
observation, not something the fixture rewrite could or needed to address):
`canary_scan()` only ever runs against the **assembled build-output staging
directory**, never the source tree directly. `skills/_parallax/scripts/` is
deliberately excluded from both the plugin and web bundling paths as
author-time repo tooling. That exclusion is correct for its stated purpose, but
it means this whole directory — including its test file — is invisible to the
one automated mechanism in this repo built specifically to catch this leak
class. See "Non-blocking structural observation" below.

### Remediation applied

Verified directly by reading the current working-tree file
(`skills/_parallax/scripts/test_build_bundle.py`):
- `test_canary_scan_catches_case_variant_leaks` no longer hardcodes any fixture
  string. It now iterates `bb.load_canary_terms()` at runtime and plants
  `.lower()`/`.upper()`/`.swapcase()` variants, asserting `bb.canary_scan`
  raises for each — same behavioral coverage (case-insensitive matching catches
  real-shaped leaks), zero literals.
- `test_canary_allowlist_does_not_mask_sibling_identifiers` now constructs
  sibling identifiers (`f"{allowed}_raw"`, `f"{allowed}_internal"`,
  `f"{allowed}Override"`) from `bb.CANARY_ALLOWLIST` at runtime, guarded by a
  check that the allowlist entry actually overlaps a loaded scan term, plus an
  explicit `assert checked, "... test is vacuous"` so a future allowlist change
  that stops overlapping any term fails loud instead of silently passing
  nothing.
- `test_canary_scan_allows_public_contract_fields` now iterates
  `bb.CANARY_ALLOWLIST` instead of naming the field directly.
- `test_canary_allowlist_masking_cannot_manufacture_a_hit` (the
  splice/manufactured-hit edge case) was **deleted**, not rewritten. Rationale
  given: with the current single allowlist entry, any string that would
  exercise the splice condition also trips a term directly, so the deleted
  test's `pytest.raises(BuildError)` was asserting the correct outcome for the
  wrong mechanism (the direct match, not the splice). This is a real gap: the
  over-masking (splice-manufactures-a-false-hit) code path in the masking
  function's own docstring is no longer covered by any test. Not a security
  regression by itself — the masking code is unchanged — but a test-coverage
  regression on the security-relevant edge case. Recommend re-adding coverage
  the same way the other three tests were fixed: derive both the allowlist
  entry and a synthetic neighbor term from a test-local, non-tracked
  construction (e.g., monkeypatch `bb.CANARY_ALLOWLIST`/`bb.CANARY_TERMS` to a
  fictitious pair inside the test) rather than depending on today's single
  real allowlist entry's specific shape.
- Checked for the claimed "logged as deferred debt" record: grepped
  `DECISIONS.md`, `notes/LESSONS.md`, and both `build_bundle.py` and
  `test_build_bundle.py` for `TODO`/`deferred`/`debt`/`splice`/`manufactur` —
  **no entry found**. This claim could not be independently confirmed. Not
  blocking (the underlying code is unchanged and safe either way), but noting
  the discrepancy rather than passing it through unverified — recommend
  actually logging it (a one-line `DECISIONS.md` or inline `TODO` next to
  `CANARY_ALLOWLIST` pointing at the missing coverage) if that's the intent.

### Non-blocking structural observation (unchanged by this remediation)

`skills/_parallax/scripts/` is tracked and public but exempt from the
build-artifact canary gates (`canary_scan` only runs on assembled `plugin/`/
`.skill` output, and this directory is deliberately never bundled). Nothing
today automatically scans it. A naive fix — pointing `canary_scan` at the
whole tracked source tree, or specifically at `skills/_parallax/scripts/` —
would immediately self-trip: `build_bundle.py` itself contains the literal
`CANARY_TERMS` list (by necessity, that's where the terms are defined) plus
the `_BRANDING_CANARIES` codepoint assembly, so a full-tree scan needs a
scoped exemption for the term-list source file(s) themselves, not just for
build tooling generally. This is recorded here as an open, non-blocking
item — the current remediation (fixing the one instance) is sufficient to
clear this audit cycle's Critical finding; the structural gap is a
follow-up, not a re-opened FAIL.
